## 0. 문서 구조

이 문서는 세 층위로 구성된다.

| 층위 | 무엇을 정의하는가 | 섹션 |
|---|---|---|
| **사용자 흐름 (User Flow)** | 사용자가 목표를 달성하기 위해 거치는 단계 | §1 |

## 1. 사용자 흐름 (User Flow)

### 1-1. 페르소나 요약

| 페르소나 | 주 목적 | 사용 빈도 |
|---|---|---|
| **조회자 (상사)** | 분기 현황 파악, 보고 수신 | 주 1~2회 |
| **편집자 (사업전략팀 멤버)** | 본인 KR 매주 갱신 | 매주 금요일 |
| **편집자 (분기 운영 담당)** | 새 분기 생성, 멤버 관리 | 분기 1~2회 |

### 1-2. 흐름 ① — 조회자: 분기 현황 확인

**목표**: 10초 이내에 분기 진행률·위험 KR·전체 신뢰도 파악

```
[Start] URL 접속
   │
   ▼
[1] 분기 메타 바 인식 (분기명, 경과율, 갱신율)
   │
   ▼
[2] Hero 카드 4종 스캔 (진척도/달성/신뢰도/경과율)
   │
   ├─ 색이 모두 🟢 → "안정" 판단 → [End: 만족]
   │
   ├─ 🟡/🔴 색이 보임 → [3] 해당 Objective 섹션으로 스크롤
   │     │
   │     ▼
   │  [4] KR 카드 좌측 띠 색상으로 위험 KR 식별
   │     │
   │     ▼
   │  [5] KR 본문(목표·현재값·진척도·신뢰도) 탐독
   │     │
   │     ▼
   │  [End: 위험 인지]
   │
   └─ 보고 필요 → [6] 📄 PDF 다운로드 → [End: 보고서 저장]
```

### 1-3. 흐름 ② — 편집자: 매주 금요일 갱신

**목표**: 3분 이내에 본인 담당 KR 갱신 완료

```
[Start] 매주 금요일 알림 (Slack 등 외부 알림 [가정])
   │
   ▼
[1] URL 접속
   │
   ▼
[2] 분기 메타 바에서 "갱신 N/M" 확인 (본인 미갱신 인지)
   │
   ▼
[3] 본인 담당 KR 카드 탐색 (Owner 칩에 본인 이름 있는 카드)
   │
   ▼
[4] KR 카드 클릭 → 인라인 편집 모드 진입
   │
   ▼
[5] 입력
   ├─ 현재값 (숫자)
   ├─ 현재값 상세 (텍스트 메모, 예: "IBK-다날, SBVA")
   ├─ 진척도 (% 정수)
   └─ 신뢰도 (高/中/低)
   │
   ▼
[6] [저장] 클릭
   │
   ├─ 성공
   │  ├─ 토스트 "저장 완료" (우상단)
   │  ├─ 갱신일 자동 기록
   │  ├─ Hero 카드 즉시 갱신
   │  └─ KR 카드 색상·상태 배지 즉시 갱신
   │  │
   │  ▼
   │  [7] 본인의 다른 KR이 남았는가?
   │     ├─ 예 → [3]으로
   │     └─ 아니오 → [End: 갱신 완료]
   │
   └─ 실패 (네트워크 오류 등)
      ├─ 토스트 "저장 실패. 다시 시도하세요"
      ├─ 입력값 폼에 유지
      └─ [6]으로 재시도
```

### 1-4. 흐름 ③ — 편집자: 분기 종료 회고

**목표**: 분기 종료 시 점수표·KPT 작성 → PDF 보고서 출력

```
[Start] 분기 종료 직전 (예: 6/22 이후)
   │
   ▼
[1] URL 접속
   │
   ▼
[2] 하단으로 스크롤 → 점수표 섹션 [E]
   │
   ▼
[3] KR별 행에 "결과" "Commit/Aspire 평가" 입력
   │  (각 셀 클릭 → textarea 전환 → 입력 → blur로 저장)
   │
   ▼
[4] 더 아래 스크롤 → KPT 회고 섹션 [F]
   │
   ▼
[5] 본인 이름 카드에 Keep / Problem / Try 입력
   │
   ▼
[6] 다른 멤버들도 동일하게 본인 카드 작성 (각자 수행)
   │
   ▼
[7] 작성 완료 후 [A] 📄 PDF 다운로드 클릭
   │
   ▼
[End: 보고서 출력]
```

### 1-5. 흐름 ④ — 편집자: 새 분기 시작

**목표**: 다음 분기 빈 폼 생성, 이전 분기 자동 아카이브

```
[Start] 새 분기 시작일 도래
   │
   ▼
[1] URL 접속
   │
   ▼
[2] [A] [+ 새 분기] 클릭
   │
   ▼
[3] 모달 [M-1] 오픈
   │
   ▼
[4] 입력
   ├─ 분기명 (예: "26.Q3")
   ├─ 시작일 (예: 2026-07-01)
   └─ 종료일 (예: 2026-09-30)
   │
   ▼
[5] [생성] 클릭
   │
   ├─ 성공
   │  ├─ 26.Q2 → archived 전환
   │  ├─ 26.Q3 → active로 생성·활성화
   │  ├─ 페이지가 26.Q3로 자동 전환 (URL `/?q=26Q3`)
   │  └─ 모달 닫힘, 토스트 "새 분기 생성 완료"
   │  │
   │  ▼
   │  [6] Objective·KR 입력 시작
   │     │
   │     ▼
   │  [7] [+ Objective 추가] → 인라인 편집으로 제목 입력
   │     │
   │     ▼
   │  [8] [+ KR 추가] → 유형·Owner·목표 입력
   │     │
   │     ▼
   │  [End: 새 분기 운영 시작]
   │
   └─ 실패 (예: 분기명 중복)
      ├─ 모달 내 에러 메시지
      └─ [4]로 재입력
```

### 1-6. 흐름 ⑤ — 편집자: 멤버 관리

```
[Start] 멤버 변경 필요 (입사/퇴사/조직 변경)
   │
   ▼
[1] [A] [👥 멤버 관리] 클릭
   │
   ▼
[2] 모달 [M-2] 오픈
   │
   ▼
[3] 분기
   ├─ 추가: 입력란에 이름 입력 → [추가] → 즉시 반영
   │
   └─ 삭제: 멤버 옆 ✕ → 확인 다이얼로그 [M-3]
        ├─ 취소 → 모달 유지
        └─ 삭제 → 즉시 반영, KR Owner에서도 제거됨 [가정]
   │
   ▼
[4] 모달 닫기
   │
   ▼
[End: KR Owner 드롭다운·KPT 카드에 변경 반영]
```

### 1-7. 흐름 ⑥ — 조회자/편집자: 과거 분기 조회

```
[Start] 과거 분기 데이터 필요
   │
   ▼
[1] [A] Quarter Selector ▼ 클릭
   │
   ▼
[2] 분기 목록 표시 (활성 = 굵게, 아카이브 = 회색)
   │
   ▼
[3] 과거 분기 선택 (예: 26.Q1)
   │
   ▼
[4] URL `/?q=26Q1`로 변경, 데이터 교체
   │
   ▼
[5] 화면 변화
   ├─ 헤더에 "읽기 전용" 배지 노출
   ├─ 모든 편집·삭제·추가 버튼 비활성화
   └─ 인라인 편집 진입 불가
   │
   ▼
[End: 데이터 조회 또는 PDF 다운로드]
```

## 2. 화면 흐름 (Screen Flow)

### 2-1. 전체 화면 맵

```
        ┌──────────────────────────────┐
        │   메인 페이지 (단일 라우트)   │
        │   /  또는  /?q={qid}         │
        └──────────────────────────────┘
              │     │      │      │
              │     │      │      │
         [+ 새 분기] │ [👥 멤버 관리] [📄 PDF]
              │     │      │      │
              ▼     │      ▼      ▼
        ┌─────────┐ │ ┌─────────┐  브라우저
        │ M-1     │ │ │ M-2     │  인쇄/저장
        │ 새 분기  │ │ │ 멤버 관리│  다이얼로그
        │ 생성    │ │ │         │
        └────┬────┘ │ └────┬────┘
             │      │      │
             ▼      │      ▼
        새 분기 활성 │   ┌────────┐
        URL 갱신    │   │  M-3   │
                    │   │ 삭제   │
            [✕ 삭제]│   │ 확인   │
                    │   └────────┘
                    │
                    ▼
              Quarter Selector
              → URL 변경, 데이터 교체
```

### 2-2. 메인 페이지 진입 흐름

```
[URL 접속]
   │
   ▼
[데이터 로드 시작 — 로딩 상태]
   │
   ├─ 분기 데이터 있음
   │  │
   │  ▼
   │  [화면 렌더링]
   │  ├─ 활성 분기 → 전체 편집 가능
   │  └─ 아카이브 분기 → 읽기 전용 UI
   │
   └─ 분기 데이터 없음
      │
      ▼
      [Empty State 화면]
      └─ "[+ 새 분기 만들기]" 단일 CTA만 노출
```

### 2-3. 모달 진입/종료 흐름

모든 모달은 공통 패턴을 따른다.

```
[트리거 액션] (버튼 클릭 등)
   │
   ▼
[배경 딤 + 모달 슬라이드 인]
   │
   ▼
[사용자 입력 또는 확인]
   │
   ├─ [확인/저장] 클릭
   │  ├─ 성공 → 모달 닫힘, 토스트 노출, 화면 갱신
   │  └─ 실패 → 모달 유지, 에러 메시지 노출
   │
   ├─ [취소] 또는 배경 클릭
   │  └─ 모달 닫힘, 입력값 폐기
   │
   └─ ESC 키 → [취소]와 동일
```

### 2-4. URL/분기 전환 흐름

```
[Quarter Selector ▼ 선택]
   │
   ▼
[URL 파라미터 갱신]  /?q={선택분기ID}
   │
   ▼
[데이터 페치 — 로딩 상태]
   │
   ▼
[화면 콘텐츠 교체]
   ├─ [A] 헤더 액션 버튼: 활성 분기면 enabled, 아카이브면 disabled
   ├─ [B] 메타 바: 새 분기 메타로 교체
   ├─ [C] Hero: 새 분기 데이터 기반 재계산
   ├─ [D] OKR 트리: 새 분기 트리로 교체

   ▼

### 2-5. 인라인 편집 흐름 (KR 카드 기준)

```
[KR 카드 클릭]
   │
   ├─ 활성 분기 → 편집 모드 진입
   │  │
   │  ▼
   │  [카드 본문이 입력 폼으로 전환]

## 3. 상태 전이 (State Transition)

### 3-1. 분기 (Quarter) 상태

```
       ┌──────────────┐
       │  존재하지     │
       │  않음 (None) │
       └──────┬───────┘
              │ [+ 새 분기] 생성
              ▼
       ┌──────────────┐
       │   active     │◄──────┐
       │  (활성, 편집 │       │ (해당 없음 — 한 번 archive되면 복귀 없음)
       │   가능)      │       │
       └──────┬───────┘       │
              │ 새 분기 생성   │
              ▼               │
       ┌──────────────┐       │
       │  archived    │───────┘
       │ (아카이브,    │
       │  읽기 전용)   │
       └──────────────┘
```

| From | Event | To | 부수 효과 |
|---|---|---|---|
| None | [+ 새 분기] 생성 성공 | active | URL이 새 분기 ID로 갱신 |
| active | 다른 분기 [+ 새 분기] 생성 | archived | 이전 active 분기가 archived로, 새 분기가 active로 |
| archived | (없음 — 되돌릴 수 없음) | - | - |

> **불변식**: 동시에 1개의 분기만 active 상태일 수 있다.

### 3-2. KR 표시 상태

진척도와 신뢰도 조합으로 자동 결정. 데이터 자체의 상태가 아니라 **표시 상태**임에 유의.

```
                  ┌──────────────────────────┐
                  │   미시작 (Not Started)   │
                  │   진척도=0 AND 갱신=없음 │
                  └────────────┬─────────────┘
                               │ 첫 갱신
                               ▼
            ┌──────────────────────────────────┐
            │      진행 중 (In Progress)        │
            │  진척도 < 달성 임계값 AND          │
            │  (진척도 ≥ 30%) AND (신뢰도 ≠ 低) │
            └──┬─────────────────┬─────────────┘
               │                 │
               │ 진척도<30%       │ 진척도가 달성
               │ OR 신뢰도=低     │ 임계값 도달
               ▼                 ▼
        ┌─────────────┐    ┌─────────────┐
        │  위험 (Risk) │    │ 달성 (Done) │
        │             │    │             │
        └─────┬───────┘    └─────┬───────┘
              │ 진척도 회복         │ (계속 갱신 가능,
              │ AND 신뢰도 회복     │  배지는 유지)
              ▼                    │
        진행 중으로 복귀             │
                                   │
              ┌────────────────────┘
              │ 우선순위 규칙: 달성 > 위험
              ▼ (달성 후엔 신뢰도 低여도 위험 배지 표시 안 함)
```

| From | 조건 | To |
|---|---|---|
| 미시작 | 첫 저장 발생 (진척도 또는 신뢰도 입력) | 진행 중 / 위험 / 달성 (조건에 따라) |
| 진행 중 | 진척도 ≥ 달성 임계값 (Commit 1.0 / Aspire 0.6) | 달성 |
| 진행 중 | 진척도 < 30% OR 신뢰도 = 低 | 위험 |
| 위험 | 진척도 ≥ 30% AND 신뢰도 ≠ 低 (달성 임계값 미만) | 진행 중 |
| 위험 | 진척도 ≥ 달성 임계값 | 달성 |
| 달성 | (진척도가 달성 임계값 미만으로 떨어짐) | 진행 중 또는 위험 [확인 필요] |

> **[확인 필요]**: 달성 후 진척도가 다시 떨어지면 라벨을 떼야 하는가? 단순화 권장 = 한 번 달성하면 라벨 유지. 아래 §6에 옵션 정리.

### 3-3. KR 카드 UI 상태 (보기 ↔ 편집)

```
       ┌──────────────┐
       │   보기 (View) │◄──────────────────┐
       └──────┬───────┘                   │
              │ 카드 클릭 (활성 분기)       │
              ▼                            │
       ┌──────────────┐                   │
       │  편집 (Edit)  │                   │
       └──────┬───────┘                   │
              │                            │
       ┌──────┴──────┐                    │
       │             │                     │
   [저장] 클릭     [취소] 클릭             │
       │             │                     │
       ▼             ▼                     │
  ┌──────────┐  입력값 폐기                │
  │ 저장 중   │       │                     │
  │ (Saving) │       └─────────────────────┘
  └────┬─────┘
       │
       ├─ 성공 → 보기 (Hero·카드 갱신, 토스트)
       └─ 실패 → 편집 (입력값 유지, 에러 토스트)
```

| From | Event | To |
|---|---|---|
| 보기 (활성 분기) | 카드 클릭 | 편집 |
| 보기 (아카이브) | 카드 클릭 | 보기 (변화 없음) |
| 편집 | [저장] 클릭 | 저장 중 |
| 편집 | [취소] 또는 ESC | 보기 (입력 폐기) |
| 저장 중 | API 성공 | 보기 (데이터 갱신) |
| 저장 중 | API 실패 | 편집 (입력 유지, 에러 표시) |

### 3-5. 회고 (점수표 / KPT) 작성 상태

각 셀(점수표의 결과·평가, KPT의 K/P/T)은 독립적으로 상태를 가진다.

| 상태 | 표시 |
|---|---|
| 빈 칸 | placeholder 텍스트로 안내 (예: "분기 종료 후 작성하세요") |
| 입력 중 | textarea 활성, 글자 카운트 노출 [가정] |
| 저장됨 | 텍스트 그대로 표시 |

상태 전이는 §3-3과 동일 패턴 (보기 → 편집 → 저장 중 → 보기).

### 3-6. 페이지 데이터 로드 상태

## 5. 시간 기반 자동 동작

사용자 액션 없이도 시간 흐름에 따라 표시가 바뀌는 항목.

| 시점 | 자동 동작 | 영향 |
|---|---|---|
| 매 페이지 진입 | 분기 메타 자동 재계산 | 경과율, 남은 주차, 현재 주차 |
| 매주 월요일 0시 | 이번 주 갱신율 카운터 리셋 | "갱신 N/M"의 분자가 0으로 |
| 분기 종료일 도래 | (자동 동작 없음) | 분기는 새 분기 생성 시점에만 archive 전환 |

> 분기 종료일 = 그 분기가 자동으로 archive되지 않음. 사용자가 다음 분기를 생성해야만 archive된다.
